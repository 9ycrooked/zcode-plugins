# 验证机制总览

本文件汇总各阶段的验证策略。各阶段 reference 里有细节,这里是全局视角。

## 为什么需要验证

LLM 生成的代码、trace、讲解都可能出错。每个阶段不加验证地放行,错误会累积到最终 content.json。
验证的核心目的:**把错误拦在产生它的阶段,不要带到下游。**

## 验证类型矩阵

| 阶段 | 产出 | 验证方式 | 失败处理 |
|------|------|---------|---------|
| 1 DAG | dag.json | **人工审核**(必须) | 用户驳回 → 重新萃取 |
| 2 代码 | code 对象 | 自动运行 + 比对 | 输出不符 → 重新生成代码 |
| 3 trace | trace 数组 | LLM 自检 + 自动比对 + 人工抽查 | 不一致 → 重新生成 trace |
| 4 讲解 | teaching 对象 | 质量自检 + 人工抽查 | 不达标 → 重新生成 |
| 5 动画 | animations | 引用合法性检查 + 人工抽查 | 引用非法 → 重新分配 |
| 6 组装 | content.json | **schema 校验**(必须) | 校验失败 → 定位字段 → 回对应阶段 |

## 三层 trace 验证(最关键)

trace 是最容易出错的环节(LLM 模拟执行可能算错变量值),用三层防御:

```
第一层:LLM 自我验证
  └─ prompt 内要求"从头检查每帧变量值,有矛盾就修正"
      ↓
第二层:自动验证
  └─ 实际运行代码,比对 trace 末帧 output 与真实 stdout
      ↓
第三层:人工抽查
  └─ 只看第二层标记"不一致"的,人工判断对错
```

详见 `stage3-trace.md`。

## 代码验证的三种情况

| verifiable | error_type | 验证方式 |
|-----------|------------|---------|
| true | (无) | 运行代码,比对 stdout == expected_output |
| false | compile_error | 编译代码,检查错误信息包含 expected_error |
| false | (无) | 跳过自动验证,直接人工审核 |

详见 `stage2-code.md`。

## schema 校验(最终防线)

阶段 6 用 `assets/schema.json` 对完整 content.json 做校验。
这是最后一道防线——即使前面验证都过了,schema 校验还能拦住结构错误。

校验失败时,**不要在组装阶段硬改**,而是定位到出错字段,回到对应阶段修。

## 审核状态流转

每个产出物有隐含状态:

```
draft(刚生成) → verified(验证通过) → approved(用户确认)
```

- 只有 `verified` 的才能进下一阶段
- 只有 `approved`(尤其阶段 1 的 DAG)才算真正定稿
- 用户随时可以驳回,回到 draft 重新生成

## 原则:宁严勿松

如果某项验证拿不准(比如 trace 变量值不确定对不对):
- 标记为"需审核",交给用户看
- 不要默认通过然后带到下游

错误带到下游的修复成本,远大于在本阶段拦住的成本。
